# Opus 모델 최적화 운영 가이드 (Claude Code 개발 세션용)

이 문서는 하네스 v1 개발을 Claude Opus 모델(Opus 4.7 이상)로 진행할 때
성능과 토큰 효율을 최대로 끌어올리기 위한 운영 규칙 + 프롬프트 모듈이다.

---

## 1. 핵심 원칙 3가지

### 1.1 Opus는 지시를 문자 그대로 따른다

Opus 4.7 이후 모델은 이전 세대보다 지시를 훨씬 문자 그대로(literal) 해석한다.
암묵적 맥락을 알아서 채워주길 기대하는 프롬프트는 성능이 떨어진다.

따라서 모든 작업 지시는 다음 4요소를 명시한다.

```text
1. 의도 (왜 하는가)
2. 제약 (하지 말 것, 범위 밖인 것)
3. 완료 기준 (무엇이 통과면 끝인가 — 우리는 acceptance test 1~5)
4. 관련 파일 경로 (어디를 보고 어디를 수정하는가)
```

### 1.2 첫 턴에 전체 작업을 명세한다

작업을 여러 턴에 걸쳐 조금씩 설명하면 턴마다 추론 오버헤드가 쌓여
토큰 효율과 품질이 모두 떨어진다.

```text
- 작업 명세는 첫 턴에 한 번에 준다.
- 질문이 필요하면 "구현 전에 질문을 한 번에 모아서 해라"라고 지시한다.
- 중간 개입을 줄일수록 Opus가 잘 작동한다.
```

### 1.3 thinking은 adaptive — 프롬프트로 조절한다

Opus 4.7+는 고정 thinking budget이 없고 adaptive thinking만 지원한다.
모델이 스스로 언제 얼마나 생각할지 결정하며, 프롬프트 문구로 조절할 수 있다.

```text
더 깊게: "이 문제는 보기보다 복잡하다. 구현 전에 단계별로 신중하게 검토해라."
더 빠르게: "깊은 분석보다 빠른 응답을 우선해라. 단순 반복 작업이다."
```

## 2. effort 레벨 운용 (Claude Code)

Opus 4.7+의 effort는 low / medium / high / xhigh / max 5단계이고
Claude Code 기본값은 xhigh다. `/effort` 명령으로 세션 중 변경 가능.

하네스 v1 개발 기준 권장:

| 작업 | effort | 이유 |
|---|---|---|
| scaffold, 파일 복사, 템플릿 생성 | low~medium | 단순 작업에 xhigh는 토큰 낭비 |
| CLI 명령/registry 로드 구현 | high | 일반 구현 |
| runWorkflow, 상태 관리, 에러 처리 | xhigh (기본) | 논리 복잡도 있음 |
| 설계 갈림길, 디버깅 막힘 | max 또는 프롬프트에 ULTRATHINK | 단일 턴 최대 추론 |

주의: 전부 max로 올리는 건 낭비다. 낮은 effort의 최신 Opus가
이전 세대의 중간 effort와 비슷한 수준이므로 작업별로 맞춘다.

## 3. 세션 운영 규칙

```text
- Explore → Plan → Confirm → Do 순서를 지킨다.
  "먼저 분석하고 계획을 제안해라. 내가 승인하기 전에 코드를 쓰지 마라."
- 무관한 작업 사이에는 /clear로 컨텍스트를 비운다. 주제 섞임 = 품질 저하.
- 긴 내용은 붙여넣지 말고 파일 경로로 참조시킨다.
- /context로 컨텍스트 사용량을 주기적으로 확인하고,
  필요 시 /compact [남길 초점]으로 압축한다.
- 작업 단위 완료 시마다 커밋시킨다. (롤백 지점 확보, /rewind 활용 가능)
- CLAUDE.md는 짧게 유지한다. 길어지면 규칙 절반이 무시된다.
  Opus가 기본으로 잘 지키는 규칙은 삭제한다.
- 병렬화가 필요하면 명시적으로 지시한다. Opus 4.7+는 기본적으로
  subagent를 남발하지 않으므로, 원할 때 "subagent N개로 병렬 진행"이라고 말한다.
```

## 4. 하네스 개발 세션에 그대로 붙일 Opus 최적화 프롬프트 모듈

아래 블록을 `claude_code_minimal_context_start_prompt.md` 뒤에 이어 붙인다.

```text
## Opus 작업 규칙

너는 Opus 모델이다. 다음 규칙으로 작동해라.

1. 지시를 문자 그대로 따른다. 명세에 없는 기능을 추측해서 추가하지 않는다.
   모호한 부분은 추측하지 말고 구현 전에 질문을 한 번에 모아서 물어라.

2. 작업 흐름: 분석 → 계획 제안 → 사용자 승인 → 구현 → 검증 → 커밋.
   승인 전에 파일을 수정하지 마라.

3. thinking 배분:
   - 단순 작업(파일 생성, 템플릿, 복사)은 빠르게 처리해라. 깊은 분석 불필요.
   - runWorkflow/에러 처리/상태 관리 설계는 구현 전에 단계별로 신중하게 검토해라.
   - 같은 오류가 2회 반복되면 멈추고, 오류를 의도적으로 재현해서 원인을 단계별로
     분석한 뒤 수정해라. 추측 수정을 반복하지 마라.

4. 토큰 절약:
   - 이미 읽은 파일을 다시 통째로 읽지 마라. 수정 대상 부분만 확인해라.
   - 응답에서 코드 전체를 반복 출력하지 말고 변경 diff 중심으로 보고해라.
   - 탐색이 필요하면 범위를 좁혀라. 열린 탐색으로 컨텍스트를 채우지 마라.

5. 검증:
   - 각 구현 단위 완료 시 해당 acceptance test를 실제 실행해서 통과를 확인해라.
   - "될 것 같다"로 완료 보고하지 마라. 실행 결과를 근거로 보고해라.
   - 완료 기준은 docs/ACCEPTANCE_TEST_CHECKLIST.md Test 1~5 전부 통과다.

6. 작업 단위 완료 시: git commit → WORKLOG.md 한 줄 기록 →
   다음 작업이 남았으면 계속, 세션 종료 시 CONTEXT_SUMMARY.md 갱신.
```

---

## 5. 세션 시작 체크리스트 (사람용)

```text
[ ] 모델이 Opus인지 확인
[ ] effort 확인 (/effort — 기본 xhigh, 단순 작업 몰린 날은 high로)
[ ] 새 세션이면 CLAUDE.md + CONTEXT_SUMMARY.md + TASKS.md만 읽혔는지 확인
[ ] 시작 프롬프트 + Opus 모듈(4번) 붙여넣기
[ ] 첫 턴에 오늘 작업 범위와 완료 기준 명시
```
