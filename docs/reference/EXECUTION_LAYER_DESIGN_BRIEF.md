# EXECUTION_LAYER_DESIGN_BRIEF.md

실행 계층(execution layer) 설계 핸드오프 문서.

- **작성일**: 2026-07-08
- **상태**: 설계 착수 전 정리 (구현 아님). 설계 자체는 별도 세션(Fable 모드)에서 진행 예정.
- **목적**: "문서를 만든 다음, 알아서 Claude Code로 작업까지 진행하는 오케스트레이터"라는 창업자 비전과, 현재 v2.6.0 하네스의 실제 구현 사이의 갭을 정리하고, 다음 설계 세션이 답해야 할 질문을 못박는다.
- **연관 문서**: [ROADMAP.md](./ROADMAP.md) v3 "실행 연결 실험", [PROVIDER_ARCHITECTURE_V2.md](./PROVIDER_ARCHITECTURE_V2.md), [PERMISSION_POLICY.md](./PERMISSION_POLICY.md), [FAILURE_RECOVERY.md](./FAILURE_RECOVERY.md)

## 1. 한 줄 요약

지금 하네스는 비전의 **"앞쪽 절반(사고·계획)"만** 구현돼 있다. 코어 에이전트가 판단·계획·비평 문서를 만들고 **거기서 멈춘다.** 그 뒤에 실제 작업을 수행하는 "**실행 절반**"(문서→자동 실행, 병행 세션, 라이브 분화)은 아직 없다. 이건 의도된 v1/v2 범위 제한이었다.

---

## 2. 창업자가 상상한 오케스트레이터 (원문 비전)

> 코어 에이전트 기반으로 일단 문서를 만들고, 그 후에 알아서 Claude Code를 통해 작업을 진행하고, 병행 세션으로 디자인도 하고, 세션 여러 개를 한 번에 열어서 작업하고, 특화 세션이 필요하면 분화해서 작업하고 — 나는 중간중간 큰 이슈만 예/아니요로 체크한다.

분해하면 6가지:

1. 코어 에이전트가 판단/계획 문서 생성
2. 그 문서를 근거로 **자동으로 Claude Code 실행**해서 실제 작업 수행
3. **병행 세션** — 개발과 디자인 등을 동시에
4. **다중 세션** 동시 오픈
5. 필요 시 **특화 세션으로 분화**
6. 창업자는 **큰 이슈만 예/아니요 게이트**로 개입

---

## 3. 현재 상태 매핑 (v2.6.0 기준)

| # | 비전 | 상태 | 근거 / 위치 |
|---|------|------|------------|
| 1 | 코어 에이전트 문서 생성 | ✅ 있음 | 워크플로우의 전부. `src/core/runWorkflow.ts`, 예: idea-validation = chief_of_staff→research→pm→red_team→founder_ceo |
| 2 | 문서 후 자동 Claude Code 실행 | ❌ 없음 | 워크플로우는 문서 저장 후 종료. 실행 다리는 **수동** — `harness task-prompt`가 붙여넣기용 프롬프트만 생성 |
| 3 | 병행 세션 (디자인 동시) | ❌ 없음 | 스텝은 순차 실행. "라이브 세션" 개념 자체가 없음 |
| 4 | 다중 세션 동시 오픈 | ❌ 없음 | 동시 실행/격리 메커니즘 없음 |
| 5 | 특화 세션 분화 | 🟡 뼈대만 | `fanout`/`--allow-spawn`으로 하위 에이전트 분화(`runWorkflow.ts` fanout 처리). 단 라이브 코딩 세션이 아니라 여전히 *문서 생성* 호출 |
| 6 | 큰 이슈만 예/아니요 게이트 | ✅ 있음 | `approval` step (y/N stdin, `src/commands/run.ts` `stdinApprover`). `gate` 조건 분기도 존재 |

핵심: **②③④가 비어 있다.** 이게 "실행 계층"이다.

## 4. 왜 여기서 멈췄나 (의도된 범위 제한)

- v1 규칙(CLAUDE.md): **"Claude Code 자동 실행 금지, Agent Teams 연동 금지, OMC 연동 금지."** MVP를 "판단·계획을 문서로 뽑는 사고 엔진"으로 못박음.
- ROADMAP에서 실행 연결은 **v3 "실행 연결 실험"** 으로 분리(착수 조건: v2로 아이디어 2개 이상 검증, 하나는 개발 착수까지).
- 즉 비전은 로드맵과 어긋난 게 아니라, **아직 도달하지 않은 뒷단계**다.

## 5. 이미 깔려 있는 재사용 가능한 뼈대

실행 계층을 새로 설계할 때 맨땅이 아니다. 아래는 그대로/확장해서 쓸 수 있는 조각들:

- **승인 게이트** (`approval` step) → 비전 ⑥ "예/아니요 체크"의 기반. 라이브 실행 흐름에 그대로 물릴 수 있음.
- **동적 분화** (`fanout` + `--allow-spawn`, `SPAWN id=... | name=... | focus=...` 선언) → 비전 ⑤의 기반. 지금은 문서 생성이지만, "분화 대상 = 라이브 세션"으로 치환하는 게 핵심 설계 포인트.
- **claude-code provider** (`src/providers/claudeCodeProvider.ts`, `claude -p --output-format json`) → 비전 ②의 첫 조각. 단 **헤드리스 1회 응답**이라, 파일을 실제로 편집하는 에이전틱 세션과는 다름.
- **진행 표시자** (`ProgressReporter`, `src/commands/progress.ts` — 2026-07-08 추가) → 비전 ②~④의 "실행 상태를 사람에게 스트리밍"의 첫 조각. 현재는 단일 스텝 스피너지만, 다중 세션 상태판으로 확장 가능한 인터페이스.
- **run_state.json + resume** (`FAILURE_RECOVERY`) → 장기 실행/중단 복구의 기반.

## 6. "실행 계층"이 실제로 요구하는 것 (설계 세션 입력)

작은 추가가 아니라 별도 프로젝트급이다. 설계에서 반드시 다뤄야 할 축:

1. **에이전틱 세션 실행 방식**
   - `claude -p`(1회 응답)로는 부족 → 파일을 실제 편집하는 에이전틱 세션 필요.
   - 후보: (a) `claude` 에이전틱 모드/headless 반복, (b) Claude Agent SDK. 트레이드오프(구독 vs API 과금, 제어권, 관측 가능성) 비교 필요. → PROVIDER_ARCHITECTURE_V2와 연결.

2. **병행/다중 세션 격리**
   - 여러 세션이 같은 레포를 동시에 편집 → 충돌.
   - 후보: **git worktree per 세션**, 또는 디렉토리/브랜치 분리. 병합 전략 필요.

3. **세션 수명 관리 + 상태 스트리밍**
   - 시작/진행/완료/실패, 사람에게 실시간 표시(스피너/상태판), 중단·재개.
   - 방금 만든 `ProgressReporter`를 다중 세션용으로 일반화하는 지점.

4. **승인 게이트를 라이브 실행에 통합**
   - "큰 이슈"의 정의(무엇이 예/아니요를 트리거하나: 파일 삭제? 외부 호출? 브랜치 병합?).
   - 비동기/병행 상황에서 게이트를 어떻게 직렬화해 사람에게 물을지.

5. **권한·안전 경계** (PERMISSION_POLICY, ROADMAP v3 보안 baseline)
   - 자동 실행 세션의 권한 최소화, 파괴적 작업 차단, "완전 무인 자동 개발"은 **여전히 금지**(모든 실행은 사람 승인 게이트 유지).

## 7. ROADMAP v3와의 관계 (조정 필요 지점)

현재 ROADMAP v3 "실행 연결 실험"은 실행 연결을 **좁게** 정의한다:

> task prompt → Claude Code 반자동 실행 (사람 승인 게이트 필수, headless mode 실험)

비전(②③④)은 이보다 넓다 — **병행/다중 라이브 세션 오케스트레이션**. 설계 세션의 산출물로 v3 정의를 이 비전에 맞게 갱신할지, 아니면 v3(반자동 단일)와 v4(병행 오케스트레이션)로 쪼갤지 결정 필요.

**유지되는 불변 원칙**(ROADMAP "v3에서도 안 하는 것):
- 완전 무인 자동 개발 금지 — 모든 실행은 사람 승인 게이트 유지.
- 외부 skill/agent 마켓플레이스 임포트 금지 (공급망 리스크).

## 8. 다음 스텝 (설계 세션이 답해야 할 질문)

Fable 모드 설계 세션에서 아래를 산출물로 만든다:

1. **실행 세션 provider 결정**: 에이전틱 `claude` vs Agent SDK — 비용/제어/관측 비교표 + 결정 (→ PROVIDER_ARCHITECTURE_V2 확장).
2. **병행 격리 모델**: worktree/브랜치/디렉토리 중 택. 병합·충돌 전략.
3. **오케스트레이션 실행 모델**: 문서(계획) → 실행 세션으로 넘기는 인터페이스. 분화 선언(`SPAWN`)을 라이브 세션으로 승격하는 방식.
4. **승인 게이트 트리거 정의**: "큰 이슈"의 구체 목록 + 병행 상황 직렬화 UX.
5. **상태 관측 모델**: 다중 세션 진행 상태판(현재 `ProgressReporter` 일반화 방향).
6. **범위 쪼개기**: v3(반자동 단일) vs v4(병행 오케스트레이션) 경계 확정.
7. **안전/권한 baseline**: 자동 실행 세션의 권한 최소화 규칙.

산출물 위치 제안: `docs/reference/EXECUTION_LAYER_ARCH.md`(설계 확정본) + ROADMAP v3/v4 갱신.
