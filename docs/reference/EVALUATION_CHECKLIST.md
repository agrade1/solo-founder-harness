# EVALUATION_CHECKLIST.md

## 1. 문서 목적

이 문서는 각 에이전트 결과와 하네스 산출물이 충분히 좋은지 평가하기 위한 기준이다.

하네스가 자동으로 문서를 생성하더라도, 결과 품질이 낮으면 실제 프로젝트 판단이 흔들린다.  
따라서 각 단계별 품질 기준을 명확히 둔다.

---

## 2. 공통 평가 기준

모든 agent output은 아래 기준을 만족해야 한다.

```text
- 판단이 명확한가
- 이유가 있는가
- 다음 행동이 1~3개로 정리되어 있는가
- 지금 하지 말아야 할 일이 있는가
- 다음 에이전트 또는 workflow 종료가 명확한가
- 사용자가 놓친 리스크가 포함되어 있는가
- 결과가 너무 길지 않은가
- Claude Code 작업으로 이어질 수 있는가
```

---

## 3. Research Agent 평가

좋은 Research 결과의 기준:

```text
- 고객이 구체적이다.
- 문제 상황이 구체적이다.
- 기존 대안이 포함되어 있다.
- 무료 대안이 포함되어 있다.
- 구매자와 사용자를 구분했다.
- 시장 크기보다 진입 가능성을 봤다.
- 돈 낼 가능성과 단순 관심을 구분했다.
- 개발 전 검증 질문이 있다.
```

나쁜 결과:

```text
- 시장이 크다는 말만 한다.
- 타겟이 너무 넓다.
- 기존 대안이 없다.
- “수요가 있을 것 같다”로 끝난다.
- 돈 낼 이유를 검증하지 않는다.
```

---

## 4. PM Agent 평가

좋은 PM 결과의 기준:

```text
- 문제 정의가 한 문장으로 명확하다.
- 첫 타겟 사용자가 좁다.
- 핵심 가치가 기능이 아니라 결과 중심이다.
- Must Have가 적다.
- Won't Have가 명확하다.
- 사용자 플로우가 짧다.
- 수동 MVP 가능성을 검토했다.
- 출시 기준과 검증 지표가 있다.
```

나쁜 결과:

```text
- 기능을 많이 늘린다.
- 로그인/결제/대시보드를 기본으로 넣는다.
- 모든 사용자를 타겟으로 한다.
- MVP가 풀서비스처럼 커진다.
```

---

## 5. UX/UI Agent 평가

좋은 UX/UI 결과의 기준:

```text
- 첫 화면에서 가치가 명확하다.
- 화면 수가 적다.
- 입력 부담이 낮다.
- 결과 화면이 핵심 가치다.
- CTA가 명확하다.
- 기본/로딩/빈/에러/성공 상태가 있다.
- 모바일과 접근성을 고려했다.
- 구현 가능한 컴포넌트 단위로 정리했다.
```

나쁜 결과:

```text
- 예쁜 디자인 중심이다.
- 화면이 너무 많다.
- 입력 항목이 많다.
- 애니메이션과 디자인 시스템을 먼저 제안한다.
- 결과 화면보다 기능 소개가 많다.
```

---

## 6. Tech Lead Agent 평가

좋은 Tech Lead 결과의 기준:

```text
- 구현 가능성 판단이 명확하다.
- 프론트/백엔드/DB/인증 필요 여부가 분명하다.
- 가장 단순한 기술 경로를 제안한다.
- 구현 순서가 작고 검증 가능하다.
- 파일/폴더 계획이 있다.
- 보안/비용/운영 리스크가 있다.
- Preflight 판단이 있다.
- 지금 하지 말아야 할 기술 작업이 있다.
```

나쁜 결과:

```text
- 최신 기술을 과하게 추천한다.
- 백엔드/DB/인증을 기본값으로 넣는다.
- 대규모 리팩터링을 권장한다.
- 구현 순서가 너무 크다.
- 보안/비용을 언급하지 않는다.
```

---

## 7. Red Team Agent 평가

좋은 Red Team 결과의 기준:

```text
- 가장 치명적인 반박이 먼저 나온다.
- 사용자가 안 쓸 이유를 찾았다.
- 돈을 안 낼 이유를 찾았다.
- 기존 대안이 이길 이유를 찾았다.
- 실패 시나리오가 현실적이다.
- 검증해야 할 가정이 명확하다.
- 리스크를 줄이는 방법이 있다.
- 그래도 진행한다면 조건이 있다.
```

나쁜 결과:

```text
- “어려울 수 있다” 수준으로 끝난다.
- 근거 없이 부정한다.
- 실행 가능한 검증 방법이 없다.
- 모든 리스크를 같은 중요도로 본다.
```

---

## 8. Founder CEO Agent 평가

좋은 CEO 결과의 기준:

```text
- 진행 / 축소 후 진행 / 추가 검증 / 보류 / 폐기 판단이 명확하다.
- 핵심 이유가 3개 이하로 정리된다.
- 가장 큰 리스크가 명확하다.
- 지금 하지 말아야 할 일이 있다.
- 다음 행동이 1~3개로 좁혀진다.
- 개발 착수 여부를 명확히 판단한다.
```

나쁜 결과:

```text
- 애매하게 “검토해보자”고 한다.
- 모든 가능성을 열어둔다.
- 결정 없이 요약만 한다.
- 개발 착수 기준이 없다.
```

---

## 9. Claude Code Task Prompt 평가

좋은 작업 지시문 기준:

```text
- 작업 목표가 한 문장으로 명확하다.
- 포함 범위와 제외 범위가 있다.
- 먼저 읽을 파일이 있다.
- 수정 예상 파일이 있다.
- 승인 필요한 작업이 명시되어 있다.
- 완료 기준이 있다.
- 패키지 설치/배포/DB 변경 금지 규칙이 있다.
```

나쁜 작업 지시문:

```text
- “전체 구현해줘”처럼 범위가 크다.
- 제외 범위가 없다.
- 파일 경로가 없다.
- 완료 기준이 없다.
- 위험 작업 승인 규칙이 없다.
```

---

## 10. 하네스 v1 산출물 평가

하네스 v1은 아래를 만족해야 한다.

```text
- init이 정상 동작한다.
- workflow가 정상 실행된다.
- 결과 markdown이 저장된다.
- CONTEXT_SUMMARY.md가 짧고 유용하다.
- Claude Code task prompt가 바로 사용할 수 있다.
- 실패 시 어떤 단계에서 실패했는지 알 수 있다.
- 사용자가 결과를 확인하고 다음 행동을 결정할 수 있다.
```

---

## 11. 평가 등급

각 산출물은 아래 등급으로 평가한다.

```text
A: 바로 사용 가능
B: 작은 수정 후 사용 가능
C: 보완 필요
D: 재생성 필요
E: 폐기
```

v1에서는 자동 등급화까지 하지 않아도 된다.  
하지만 사람이 평가할 때 이 기준을 사용한다.

---

## 12. 완료 기준

이 문서가 적용되면 다음이 가능해야 한다.

```text
- 에이전트 결과를 그대로 믿지 않고 품질을 점검할 수 있다.
- Claude Code에 넘길 작업 지시문 품질을 평가할 수 있다.
- 하네스가 생성한 문서가 실제로 쓸 만한지 판단할 수 있다.
```
